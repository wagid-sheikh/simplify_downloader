# 1. Objective

Design and implement a production-grade customer retention and multi-source lead management pipeline for laundry and dry-cleaning stores.

The system must:

1. Identify inactive customers from historical order behavior.
2. Generate structured retention and reactivation leads.
3. Integrate TumbleDry CRM leads already pulled by the existing `td_leads_sync` pipeline.
4. Integrate externally sourced leads from online/offline campaigns.
5. Convert all actionable leads into a unified master lead table.
6. Generate idiot-proof Excel workbooks for store teams.
7. Re-ingest completed Excel files from store teams.
8. Update lead lifecycle, follow-up lifecycle, suppression state, and recovery state.
9. Detect actual recovery from the `orders` table.
10. Send structured management summary email after each run.
11. Reuse existing project infrastructure and coding conventions.

This module MUST be implemented as an independent pipeline while reusing existing common helpers and system infrastructure.

Codex MUST NOT invent parallel infrastructure.

# 4. Lead Source Types

The system must support three lead source types.

All actionable leads from all sources must be converted into one unified operational lead table.

## 4.1 Source Type A — RETENTION

Generated by this new pipeline using inactive customer analysis.

Purpose:

- recover inactive customers
- identify churn reasons
- collect feedback
- reactivate LOST customers
- generate repeat business

These leads are capped.

Fresh retention lead cap:

```text
13 fresh retention leads per store per day
```

However, fresh retention leads are generated only when previous actionable workload is completed.

---

## 4.2 Source Type B — TD

TumbleDry CRM leads.

Context:

- For stores where `store_master.sync_group = 'TD'`, TumbleDry pushes leads into its CRM.
- Existing `td_leads_sync` pipeline already pulls these leads and stores them in DB.
- Currently, the owner manually copies these leads into WhatsApp groups.
- This pipeline must remove that manual gap.

Rules:

- Codex must inspect existing `td_leads_sync` implementation and tables.
- Pipeline must pull all pending/actionable TD leads from existing TD lead tables.
- TD leads must be converted into the unified master lead table.
- TD source tables must remain the raw source of truth.
- The unified lead table becomes the operational follow-up source of truth.
- TD leads are not part of the 13-lead retention cap.
- TD leads must appear in the same store workbook.
- TD leads must support the same follow-up, closure, suppression, and reporting lifecycle.

---

## 4.3 Source Type C — EXTERNAL

Externally sourced leads.

Examples:

- Meta campaign leads
- Google campaign leads
- local marketing campaign leads
- society campaign leads
- WhatsApp campaign leads
- referral leads
- walk-in inquiry leads manually captured
- owner-sourced leads

Rules:

- External leads must be stored in a structured table.
- External leads must be converted into the unified master lead table.
- External leads are not part of the 13-lead retention cap.
- External leads must appear in the same store workbook.
- External leads are assigned to one specific store only.
- Inter-store transfer is out of scope for now.

# 6. Lead Closure Rule

A lead becomes closed only when either:

1. An actual order is created and detected in the `orders` table after lead generation/follow-up.
2. A dead-end outcome is reached.

Dead-end outcomes include:

```text
Do Not Contact
Wrong Number
Invalid Number
Shifted Location
Not Interested
Lead Stale
```

`Do Not Contact`, `Wrong Number`, and `Invalid Number` must also create suppression records.

Recovery must be confirmed from the `orders` table.

Store marking `Pickup Requested` is a hot signal, but not final recovery.

---

# 7. Store Assignment Rule

Every lead must belong to one specific store/cost center.

Rules:

- Lead workbook is generated store-wise.
- A lead appears only in the workbook for its assigned cost center.
- Inter-store transfer/reassignment is not supported in this version.
- If a customer says they belong to another area/store, staff can write it in remarks, but system will not transfer automatically.

# 9. High-Level Pipeline Flow

The single pipeline must internally perform all sub-routines.

Recommended flow:

1. Load config.
2. Initialize run logging.
3. Fetch active stores from `store_master.customer_retention_pipeline = true`.
4. Ingest returned Excel files from `inputs/customer_followup/`.
5. Normalize and validate returned data.
6. Update unified lead table and lead history.
7. Apply suppression/DND/dead-end rules.
8. Detect recoveries from `orders`.
9. Build/update customer retention snapshot.
10. Pull due follow-ups.
11. Pull pending carry-forward leads.
12. Generate fresh retention leads only where allowed.
13. Pull pending TD leads from existing `td_leads_sync` tables.
14. Pull pending external leads.
15. Upsert/convert TD and external leads into unified master lead table.
16. Build one workbook per store.
17. Archive processed input files.
18. Generate owner summary email.
19. Close pipeline run logging.

# 10. Customer Lifecycle Buckets

Based on actual observed repeat behavior, use the following buckets.

| Bucket  | Days Since Last Order |
| ------- | --------------------: |
| ACTIVE  |                 0–21 |
| WARM    |                22–45 |
| COOLING |                46–75 |
| DORMANT |               76–120 |
| COLD    |              121–180 |
| LOST    |                  181+ |

Rules:

- ACTIVE customers are excluded from retention lead generation.
- WARM, COOLING, DORMANT, COLD, and LOST can generate retention leads.
- LOST customers must be included and prioritized, not ignored.
- LOST customers must use a different recommended strategy.

# 12. Retention Lead Generation Rules

Fresh retention lead cap:

```text
13 per store per day
```

Initial bucket allocation target:

| Bucket  | Fresh Leads / Store / Day |
| ------- | ------------------------: |
| WARM    |                         3 |
| COOLING |                         3 |
| DORMANT |                         2 |
| COLD    |                         2 |
| LOST    |                         3 |

Important owner decision:

Fresh leads are generated only when previous actionable work is completed.

Therefore:

- pending carry-forward leads come first
- due follow-ups come first
- if a store has incomplete actionable work, do not keep adding fresh retention leads blindly
- TD and EXTERNAL leads are still included because they are source-driven and not capped by retention quota

---

# 13. Due Follow-Up Rule

If `next_followup_date` is today or overdue:

- lead must appear at the top of the workbook
- lead is over and above standard fresh retention cap
- store team must handle it first

Next Follow-up Date is mandatory for:

```text
Interested
Will Give Order Later
Pickup Requested
WhatsApp Sent
No Response
Switched Off
```

If a required next follow-up date is missing, pipeline must add two days as auto-follow-up date and report it in email summary.

---

# 14. Pending Carry-Forward Rule

If previous lead work was not completed:

- carry forward the same lead
- do not auto-expire it
- do not auto-mark stale
- do not generate unlimited fresh retention leads on top of unworked leads

Important owner decision:

Only store team can explicitly mark a lead as stale using:

```text
Lead Stale
```

Pipeline must not auto-decide staleness.

---

# 15. Priority Scoring

Priority score must be computed for retention leads.

Inputs:

- lifecycle bucket
- days since last order
- total orders
- lifetime spend
- average order value
- last order amount
- previous follow-up outcome
- recovery history
- suppression state

Lifetime sales must be computed by the pipeline from `orders`.

Codex must inspect actual amount columns in `orders` before finalizing calculation. For `orders.source_system='TumbleDry'` use `orders.net_amount` for all other `source_system` use `orders.gross_amount`.

---

# 16. Suppression Rules

| Outcome          | Suppression |
| ---------------- | ----------- |
| Do Not Contact   | Permanent   |
| Wrong Number     | Permanent   |
| Invalid Number   | Permanent   |
| Shifted Location | Permanent   |
| Not Interested   | 90 days     |

Suppressed customers must be excluded from future retention lead generation until suppression expires.

Permanent suppressions do not expire.

Suppression must apply across all source types where mobile number and store match.

# 31. Recovery Detection

Recovery must be confirmed from `orders`.

A customer is recovered when:

- same cost center
- same normalized mobile number
- order date after lead generation date or after latest follow-up assignment date

When recovered:

- mark lead as recovered
- set `is_recovered = true`
- set `recovered_at`
- store recovered order id if available
- close pending leads for same customer/store
- exclude from future retention follow-ups until customer becomes inactive again

---

# 32. TD and External Lead Closure

TD and EXTERNAL leads close when:

1. New order is detected in `orders`.
2. Store response indicates dead-end:
   - Do Not Contact
   - Wrong Number
   - Invalid Number
   - Shifted Location
   - Not Interested
   - Lead Stale

If store marks:

```text
Pickup Requested
```

Then:

- keep as high-priority follow-up until actual order appears or staff later marks dead-end.

# 39. Idempotency Requirements

Pipeline must be safe to re-run.

Idempotency requirements:

- same TD source record must not create duplicate unified lead
- same external import row must not create duplicate unified lead
- same retention customer/bucket/run must not create duplicate active lead
- same workbook upload must not duplicate history
- same recovery must not repeatedly update the same lead
- archive process must not overwrite files without deterministic handling

---

# 40. Error Handling

Pipeline should fail only for critical/systemic issues.

Examples of critical failures:

- DB unavailable
- config missing
- output directory inaccessible
- workbook generation impossible for all stores

Non-critical row-level issues:

- invalid mobile number
- unknown dropdown value
- missing optional field
- edited protected column
- duplicate row

Non-critical issues must be logged and summarized, not crash the whole run.

---

# 41. Security Requirements

- No credentials in code.
- No raw SQL string concatenation with user input.
- Validate imported files.
- Protect generated workbook.
- Preserve audit history.
- Avoid leaking other store data into wrong workbook.
- No inter-store lead transfer in this version.
